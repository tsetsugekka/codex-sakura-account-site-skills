# Static Deploy Refresh Check

## Goal

Make static pages pick up newly published hashed JS/CSS after a deploy without damaging live server state. The page should quietly detect that the live entry HTML now references different same-origin JS/CSS assets, then reload itself once with a version query parameter. Successful, unchanged pages should continue using normal browser cache after HTML revalidation.

Use this skill when:

- publishing a new page whose styles or script may be cached by existing visitors,
- creating a new static/Vite page that will be deployed with hashed `assets/*`,
- batch updating older pages to include the deploy refresh behavior,
- adding old hashed asset cleanup to Sakura static deploys,
- protecting live JSON/data generated by cron or server scripts,
- merging HTML regions that are updated on the server before publishing,
- fixing reports such as "new page published but old CSS still appears" or "users must hard refresh".

## Required Behavior

- Add HTML revalidation metadata to each entry page:
  - `Cache-Control: no-cache, must-revalidate`
  - `Pragma: no-cache`
  - `Expires: 0`
- Inject a small inline deploy refresh script into entry HTML before `</head>`.
- Compare only same-origin `.js` and `.css` references from the current document and freshly fetched entry HTML.
- Fetch the live entry HTML with `cache: 'no-store'` so the check itself bypasses stale HTML caches.
- Refresh at most once per target asset signature using `sessionStorage`.
- Add a version query parameter such as `__deploy_v=<short-hash>` when refreshing.
- Keep production data files, JSON generated by cron, and scraper-owned SEO blocks untouched.
- Upload new assets before uploading the new entry HTML.
- After verifying the live page, clean old hashed assets so only the current deploy generation and one previous generation remain.
- If the live entry HTML contains server-generated or cron-injected regions, fetch the latest live HTML and merge those regions before writing the deploy HTML.

Do not use `no-store` as the default page HTML caching policy. The normal page should revalidate HTML, not disable caching for every visit.

## Sakura Deploy Safety

When this skill is used for Sakura-hosted static pages, treat deploy as a whitelist operation:

- Upload only the page entry HTML, the current hashed `assets/*`, and explicitly approved static extras.
- Exclude live data by default, including cron outputs such as `data.json`, per-day data folders, account snapshots, generated narratives, market history, and server-side cache files.
- Keep each page's cleanup scoped to that page's own `assets/` directory.
- Delete only older hashed JS/CSS/assets after the new page has been uploaded and verified.
- Keep the assets referenced by the current live HTML plus one immediately previous deploy generation for rollback.
- Run cleanup in dry-run mode first. Apply deletion only after confirming the delete list contains older hash assets and no production data.

If no cleanup script exists, add or patch a project-specific cleanup helper. It should list the remote page asset directory, identify the current live HTML asset references, group remote hashed assets by deploy time or manifest, and delete only generations older than current plus previous.

## Cron-Managed HTML And Data

Some entry HTML files are partly maintained by cron or server scripts, for example SEO `<noscript>` blocks, static rankings, status banners, generated timestamps, or other marked regions. Do not overwrite these regions from a stale local build.

Use this rule:

1. Identify server-managed regions by stable comments or known markers.
2. Before publishing, fetch the current live HTML from Sakura.
3. Merge the server-managed regions from live HTML into the new deploy HTML.
4. Then inject or preserve the deploy refresh check.
5. If the region cannot be found or parsed, stop and report the merge blocker instead of overwriting the live region.

Never solve a stale-style issue by uploading the whole build directory or replacing live data files.

## Existing Pages

1. Find entry HTML files and deploy-preparation scripts:
   - static pages: `index.html`, route aliases, extensionless HTML aliases,
   - build output: `dist/index.html`,
   - deploy scripts that copy `dist/index.html` into `_deploy/` or another staging directory.
2. Prefer adding the refresh injection to the deploy-preparation stage. This avoids editing generated `dist/` files by hand and ensures every future build keeps the behavior.
3. If a page has no deploy script, inject directly into source HTML or the published HTML template.
4. Use `scripts/inject_deploy_refresh.py` for direct HTML batch work:
   - dry-run first,
   - apply with `--write` only after reviewing the file list.
5. Verify the marker exists exactly once per entry:
   - default marker: `<!-- DEPLOY_REFRESH_CHECK_V1 -->`.

Example:

```bash
python3 skills/static-deploy-refresh-check/scripts/inject_deploy_refresh.py --include-extensionless path/to/page/index.html path/to/page/AI
python3 skills/static-deploy-refresh-check/scripts/inject_deploy_refresh.py --write --include-extensionless path/to/page/index.html path/to/page/AI
```

## New Pages

When creating a new static page or Vite app, add this skill's behavior as part of the page template or deploy workflow:

1. Put the HTML revalidation meta tags in the source `index.html`.
2. Ensure the deploy-preparation script injects the deploy refresh script into the staged HTML.
3. If the app generates route aliases, inject the same marker into each alias entry.
4. Make the upload manifest put `assets/*` first and `index.html` last.
5. Add a post-deploy check that opens the page and confirms the new asset filenames are referenced.

If a repository has a central page registry for deploy targets, add the new page there and reuse the same injection helper for all registered pages.

## Deploy Script Pattern

Use the reference pattern when patching an existing JavaScript deploy helper:

- Read `references/deploy-refresh-pattern.md` when modifying a deploy script or explaining the mechanism.
- Add a function similar to `injectDeployRefreshCheck(html)`.
- Call it when writing staged `index.html` and every alias entry.
- Make the function idempotent by checking the marker first.

If the deploy script already has an equivalent function, keep the local naming and style. Do not introduce a second independent refresh script.

## Batch Retrofit Checklist

Use this order for old pages:

1. Inventory entry pages and generated aliases.
2. Identify whether each page is source HTML, build output, or deploy-staged HTML.
3. Patch the shared deploy script first if one exists.
4. Run dry-run injection on direct HTML targets.
5. Apply injection with `--write`.
6. Build or regenerate deploy staging.
7. Merge any cron-managed HTML regions from the current live server copy.
8. Confirm each target has one marker and the three revalidation meta tags.
9. Deploy assets first, then entry HTML.
10. Verify the live page loads and references the new asset hashes.
11. Dry-run old asset cleanup for that page.
12. Apply cleanup only if it keeps current plus previous generations and deletes only older hashed assets.
13. Test a stale-page scenario if practical: open old page, deploy new asset hash, wait for the check, confirm one reload with `__deploy_v`.

## Safety Rules

- Do not overwrite production JSON or cron/scraper output while solving a cache problem.
- Do not delete old hashed assets until the live page is verified; keep at least the current and previous deploy generations when cleanup is available.
- Do not overwrite cron-injected HTML regions from stale local templates; fetch live HTML and merge those regions first.
- Do not run global asset cleanup when only one page changed.
- Do not reload in a loop. Store the attempted version in `sessionStorage`.
- Do not compare cross-origin scripts, analytics tags, CDN CSS, or third-party assets.
- Do not force refresh all pages globally when only one page was updated.
- Do not change service worker behavior unless the project already has a service worker and the issue is proven to involve it.
